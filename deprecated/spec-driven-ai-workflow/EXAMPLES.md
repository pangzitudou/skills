# Examples

## First Use

User:

```text
我第一次用这个 skill，不知道该怎么开始。
```

Coach:

```text
当前阶段：首次使用引导。

为什么是这个阶段：先判断你的工作类型和卡点，再决定进入 Brainstorm、Grill Me、Prototype、SPEC、Plan、Implementation、Human QA、Acceptance，还是公共规范维护。不要一上来就让 AI 写方案。

下一步动作：先回答一个问题。

可复制 prompt：
此时不需要执行线程。

拿到产物后：
你还没有产物。请先告诉我：你现在想推进的是哪类交付工作：新需求、重构前澄清、AI 产物验收、流程优化、SPEC 编写/审查，还是公共规范维护？
```

## Quick HTML Prototype

User:

```text
这个需求涉及一个操作页面，但我不擅长判断界面和流程。
```

Coach:

```text
当前阶段：Prototype。

为什么是这个阶段：涉及界面、流程、用户操作和状态变化时，文字容易让每个人想象不同。快速 HTML 原型是需求验证物，不是生产实现。

下一步动作：开一个执行线程生成单文件 HTML 原型，验证流程、信息架构、关键状态和业务验收点。

可复制 prompt：
请基于下面需求生成一个快速 HTML 原型，用于验证需求和交互流程。

注意：
- 这不是正式工程实现，不要选技术栈，不要拆工程任务。
- 优先做成单个自包含 HTML 文件。
- 重点验证流程、信息架构、页面结构、关键状态和业务验收点。
- 不要加入非核心的埋点、性能优化、复杂动画或过度视觉装饰。

需求目标：[填写]
目标用户/使用者：[填写]
核心流程：[填写]
关键数据或样例：[填写]
权限/异常/空状态：[填写]
我现在不确定的地方：[填写]

请输出：
1. 原型覆盖的用户路径
2. HTML 原型代码
3. 原型中体现的关键状态
4. 仍然需要业务方确认的问题
5. 哪些内容不应该在此阶段加入

拿到产物后：
真实跑一遍原型，把已确认、未确认、应删除或延后的内容带回来。不要直接进入实现。
```

## After Prototype

User:

```text
prototype 生成完成后，我应该做什么？
```

Coach:

```text
当前阶段：Prototype Review。

为什么是这个阶段：原型完成不代表需求完成。现在要验证流程、状态、信息结构和业务判断，并决定是否进入 SPEC、继续改原型、进入 Plan，或做公共规范影响分析。

下一步动作：让执行线程做原型复盘，不要写技术方案。

可复制 prompt：
请基于这个 prototype 做一次原型复盘，不要写技术方案，不要拆任务，不要进入实现。

原型材料：
[粘贴链接/截图/HTML/说明]

请输出：
1. 原型覆盖的核心用户路径
2. 已经被原型验证成立的需求点
3. 原型暴露出的未确认问题
4. 可以删除或延后的非核心内容
5. 需要补充验证的关键状态
6. 哪些结论可以进入 SPEC
7. 哪些内容还不能进入 Plan
8. 是否需要做公司级/公共 SPEC 影响分析；如果需要，只说明影响领域和原因，不要直接写完整公共规范

拿到产物后：
检查它有没有明确区分已确认、未确认、应删除/延后。只有已确认内容能进入 SPEC 或 Plan。
```

## SPEC Boundary

User:

```text
后端还没有 SPEC，要如何生成 SPEC？
```

Coach:

```text
当前阶段：SPEC。

为什么是这个阶段：后端开发前缺少合同。SPEC 应该定义目标、角色、范围、流程、规则、边界和验收，不应该展开成接口字段表、数据库表结构或实现方案。

下一步动作：让执行线程生成后端 SPEC 草稿，但严格限制在合同层。

可复制 prompt：
请为当前后端能力生成一份 SPEC 草稿。

注意：
- SPEC 是合同，不是 README、技术方案、API 文档、数据库文档、测试文档或部署文档。
- 请只定义目标、角色、范围、流程、业务规则、权限/数据边界和验收标准。
- 不要写启动命令、完整 API 字段、数据库表结构、代码实现路径、测试命令、部署命令或临时讨论。

输入材料：
1. 业务目标：[填写]
2. prototype / 前端状态材料：[填写，没有则写无]
3. 用户角色：[填写]
4. 必须支持的业务流程：[填写]
5. 已知不做范围：[填写]
6. 权限、安全、数据边界：[填写]
7. 相关公共 SPEC：[填写，没有则写无]

请输出：
1. 背景
2. 目标
3. 非目标
4. 用户角色
5. 核心场景和业务流程
6. 功能规格
7. 业务规则
8. 权限与安全
9. 数据对象与数据边界摘要
10. API 能力要求摘要
11. 非功能要求
12. 验收标准
13. 不允许做的事
14. 需要另行维护的文档引用

拿到产物后：
检查它像不像合同。如果它开始写接口字段、表结构、代码路径、测试命令或部署步骤，就让它移到对应文档。
```

## Common SPEC Impact

User:

```text
prototype 之后有些规则可能后续复用，要不要沉淀成通用规范？
```

Coach:

```text
当前阶段：公共规范影响分析。

为什么是这个阶段：项目/功能 SPEC 和公司级/公共 SPEC 不是一回事。现在要先判断哪些是可复用规则，哪些只是本次业务细节。

下一步动作：先做影响分析，不要直接写完整公共规范。

可复制 prompt：
请判断下面内容是否值得沉淀为公司级/公共 SPEC。

注意：
- 公司级/公共 SPEC 只沉淀可复用规则，不沉淀单次需求细节。
- 不要写完整规范，不要写技术方案，不要拆任务。

输入材料：
1. 当前项目/功能 SPEC：[粘贴]
2. prototype / QA / 实现发现：[填写]
3. 已有公共 SPEC 清单：[填写]
4. 已确认的业务结论：[填写]

请输出：
1. 可复用规则有哪些
2. 本次特例有哪些，不能进入公共 SPEC
3. 应该更新哪个已有 SPEC，还是新增 SPEC
4. 最小变更范围
5. 哪些详细内容应留在项目 SPEC、API、DATA_MODEL、TESTING 或 README
6. 项目 AGENTS.md / CLAUDE.md / README.md 是否需要补引用
7. 审查清单

拿到产物后：
检查它有没有把单次业务需求写成公司级规范。如果有，退回重写。
```

## Review Project Docs Against comm

User:

```text
一个项目已经接入公共 SPEC 了，我怎么检查文档体系是否合格？
```

Coach:

```text
当前阶段：公共 SPEC 接入检查。

为什么是这个阶段：现在要检查项目入口是否正确引用公共规范，以及项目文档是否避免重复、失效和私有漂移。

下一步动作：让执行线程检查根 README、docs/README、AGENTS/CLAUDE 和 API 权威文档。

可复制 prompt：
请检查这个项目是否正确接入公共 SPEC。

项目材料：
1. 根 README.md：[粘贴或让 AI 读取]
2. docs/README.md：[粘贴或让 AI 读取]
3. AGENTS.md / CLAUDE.md：[粘贴或让 AI 读取]
4. API / INTEGRATION / TESTING 等关键文档：[粘贴清单或让 AI 读取]
5. 公共 SPEC 清单或标准地图：[粘贴或让 AI 读取]

请输出：
1. 根 README 是否只承担项目定位、运行、测试、文档入口
2. docs/README 是否承担角色阅读顺序、职责矩阵、维护规则
3. AGENTS.md / CLAUDE.md 是否正确引用公共 SPEC 和项目特殊约束
4. API 字段、错误码、鉴权规则是否只有一个权威文档
5. 是否存在重复长表、临时说明长期并存、聊天记录保存关键约束
6. 是否存在失效链接或失效公共 SPEC 引用
7. 最小修复建议

拿到产物后：
只接受最小修复建议。不要让它借检查机会重写所有文档。
```

## Starting a New Requirement

User:

```text
我要开始一个新的需求，第一步做什么？
```

Coach:

```text
当前阶段：Brainstorm 前。

为什么是这个阶段：新需求不要直接进入实现，也不要先定技术方案。先从原始想法发散，明确可能方向，再用 Grill Me 收敛决策。

下一步动作：开一个执行线程做 Brainstorm。

可复制 prompt：
请和我一起 Brainstorm 这个需求。先不要写 PRD，也不要给技术方案。

背景是：[填写背景]
我想达成的目标是：[填写目标]
我现在不确定的是：[填写不确定点]

请帮我发散 3-5 个可能方向，说明每个方向适合什么场景、风险是什么、下一步应该追问什么。

拿到产物后：
把 Brainstorm 结果带回来，我帮你判断是否可以进入 Grill Me。
```

## Technology Preference Bias

User:

```text
这个老项目是 Vue，我想让 AI 继续按 Vue 重构，怎么写 prompt？
```

Coach:

```text
当前阶段：Plan 前的约束澄清。

为什么是这个阶段：以需求为中心，不以技术偏好或既有实现为中心。先判断 Vue 是真实约束，还是历史惯性。

下一步动作：先让执行线程识别真实约束，再比较沿用 Vue 与重新选型的成本、风险和收益。

可复制 prompt：
请先不要给技术方案。请基于下面需求，判断“沿用 Vue”到底是硬约束、软约束，还是历史惯性。

需求目标：[填写]
现有系统情况：[填写]
团队维护能力：[填写]
上线/迁移限制：[填写]
适用的项目 SPEC、公共 SPEC 或项目约束：[填写，没有则写无]

请输出：
1. 哪些约束是真实约束
2. 哪些只是偏好或历史惯性
3. 如果不预设技术栈，理想方案应先考虑哪些标准
4. 是否可以进入 Plan 阶段

拿到产物后：
检查它有没有把历史实现当成唯一方案。
```

## Checking a SPEC

User:

```text
AI 写了一份 SPEC，我怎么判断能不能进入 Plan？
```

Coach:

```text
当前阶段：SPEC 到 Plan 的检查点。

为什么是这个阶段：Plan 应该发生在 SPEC 合格之后。目标、角色、范围、规则、边界和验收标准不清楚时，直接做技术方案会放大返工。

下一步动作：用 SPEC 合同检查清单审查，不要让 AI 直接写技术方案。

可复制 prompt：
请基于下面这份 SPEC 做进入 Plan 前审查。

只检查需求是否清晰，不要写技术方案。
重点检查：目标、用户角色、范围、非目标、业务流程、业务规则、权限/数据边界、验收标准、适用的公共 SPEC、是否存在技术偏好或现有实现污染。

SPEC：
[粘贴]

请输出：可以进入 Plan / 需要回到 Grill Me / 需要回到 Brainstorm / 需要补原型，并说明原因。

拿到产物后：
如果它给出技术方案而不是审查结论，这个输出不合格。
```

## Human QA

User:

```text
AI 说功能做完了，我现在应该怎么人工验证？
```

Coach:

```text
当前阶段：Human QA。

为什么是这个阶段：AI 测试不能替代真实环境里的业务验收。人不是回到传统测试员角色，而是作为 AI 的手、脚、眼睛和业务判断器。

下一步动作：让执行线程基于需求和实现结果生成 QA 清单。

可复制 prompt：
请基于下面需求和实现结果生成一份人工 QA 清单。

我会作为你的手、脚和眼睛在真实环境中操作。
请告诉我每一步要点哪里、输入什么、观察什么、什么算通过、失败时要记录什么。

需求目标：[填写]
验收标准：[填写]
适用的项目 SPEC、公共 SPEC 或项目约束：[填写，没有则写无]
实现说明或变更摘要：[填写]
测试环境/地址：[填写]
已知风险或重点关注点：[填写]

请输出：
1. QA 范围和不测范围
2. 前置条件和测试数据
3. Happy path 操作步骤
4. 关键异常路径
5. 权限、空状态、错误状态、禁用状态、加载状态检查
6. 每一步的通过/失败标准
7. 失败时需要我提供的证据
8. 是否可以进入业务验收的判断标准

拿到产物后：
按步骤真实操作。发现问题时，不要只说“有 bug”，要给操作步骤、实际结果、期望结果、截图/日志、使用数据、是否可复现、是否阻塞业务验收。
```
